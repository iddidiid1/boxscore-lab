# 云端部署计划书（活文档）

> 状态：草稿 / 需求确认阶段。本文件是一份**可随时修改的活文档**，会在后续设计与实施过程中持续更新。
> 最后更新：2026-07-09

## 1. 核心目的

让"随时随地记录数据"成为可能——不依赖开着本地电脑，在手机 / 其他设备上也能记录和查看数据。（已确认值得实现）

## 2. 背景与约束

- 本项目是个人自用的数据记录应用，**单用户、数据不敏感、无分享需求**。
- 目前**没有登录 / 注册**，数据也不与任何用户绑定。
- **成本要求**：尽可能控制成本，能免费最好；免费层用量需设告警、关闭自动升级付费。

## 3. 已识别的核心问题

1. **SQLite 与临时文件系统冲突**
   - 现用 SQLite（本地文件库）。多数免费 PaaS 的文件系统是临时的，重新部署会丢数据。
   - 需在两个方向中选择：
     - A. 保留 SQLite + 挂持久卷（volume）
     - B. 迁移到托管 Postgres（更贴近真实生产，Prisma 换 provider 改动小）—— **倾向方案**
2. **无登录 → 写权限暴露**
   - 风险点不是隐私，而是**数据完整性**：任何拿到 URL 的人都能增删改数据、接口可能被扫描器爬到。
   - 结论：这不构成"不该部署"，只构成"部署时需加一层轻量门锁"。
   - 候选轻量方案：HTTP Basic Auth / 接口共享密钥 / 平台层访问限制（IP 白名单、Cloudflare Access）。
   - **不做**完整注册登录系统（超出当前需求，见 §6）。

## 4. 目前可见的基本方向

应用形态是「静态前端 + Node/Express API + 数据库」三件套，分层部署：

| 层 | 候选方案 | 备注 |
|----|---------|------|
| 前端 | Vercel（倾向）/ Cloudflare Pages | Vite 静态产物，免费额度充足，绑 GitHub 自动部署，无需容器化 |
| 后端 | Fly.io（倾向）| **自写 Dockerfile 容器化部署**；Fly.io 基于 Dockerfile，需绑卡但有免费额度（可设用量上限兜底） |
| 数据库 | Neon（倾向）/ Supabase 免费 Postgres | serverless、闲置不删库，比平台自带插件更安全 |
| 访问控制 | Basic Auth（倾向）| 轻量堵住写权限暴露 |

已有基础：前端已支持通过 `VITE_API_BASE_URL` 配置 API 地址（利于前后端分离部署）。

**容器化范围**：仅后端服务自写 Dockerfile 容器化；前端交给 Vercel、数据库交给 Neon，二者不需要容器化。迁移到 Postgres 时，可先在本地用 `docker-compose` 拉起一个 Postgres 容器做验证，不必一开始就依赖云端。

- [ ] 数据库：迁移 Postgres（倾向）还是 SQLite + 持久卷？

## 6. 明确不在范围内（当前阶段）

- 完整的用户注册 / 登录 / 多用户系统。
- 多环境（dev/staging/prod）流水线。
- 任何为"分享"或"多用户"而做的功能改造。
