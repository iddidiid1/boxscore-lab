# fantasy-league-stats — Claude Context

## Project Overview

Personal fantasy basketball web app for recording box scores and computing league stats. Intentionally simple MVP with manual data entry only — no authentication, auto-scheduling, or auto-ranking.

## Tech Stack

- **Monorepo**: pnpm workspaces
- **Frontend**: React + Vite + TypeScript — `apps/frontend/`
- **Backend**: Node.js + Express + TypeScript — `apps/backend/`
- **Database**: SQLite via Prisma — `prisma/schema.prisma`
- **ORM**: Prisma

## Key Commands

```bash
pnpm dev              # start frontend + backend concurrently
pnpm dev:frontend     # frontend only (http://localhost:5173)
pnpm dev:backend      # backend only (http://localhost:4000)
pnpm db:migrate       # run Prisma migrations
pnpm db:generate      # regenerate Prisma client
pnpm db:seed          # seed database
pnpm db:studio        # open Prisma Studio
```

## Architecture Invariants

### Never Store Computed Stats

Stats, averages, and points are **always calculated at runtime** from raw Match records. Do not add stored/persisted derivative fields.

- `totalPoints` = `BASE_TEAM_POINTS` (1000) + sum of eligible `EventResultTag.rankingPoints`
- Player averages — computed from `MatchPlayerStat`
- Team averages — computed from `MatchTeamOtherStat`

Do not add columns like `currentPoints`, `manualPoints`, `avgPoints`, `basePoints` to any model.

See `docs/TEAM_POINTS_RULES.md` and `docs/TEAM_RANKING_CONFIG.md`.

### Slug Generation

`Team.slug` and `Player.slug` are **auto-generated by the backend** from `name` at creation time using kebab-case + numeric suffix on conflict. Frontend never sends or modifies slugs. Slugs are immutable after creation.

### Deletion Policy

See `docs/DELETION_RULES.md` for full rules. Key points:

- Teams: prefer **soft delete / archive** (`archivedAt`)
- Events: the MVP application only supports archive through `archivedAt`; it exposes no Event hard-delete or restore API/UI
- Players: prefer `isActive = false`
- `Event.rankingOrder` is monotonic — never reused, renumbered, or set by frontend
- Historical match/award/stat records must never be destroyed by normal application workflows. Existing database cascades from a parent Event apply only if a controlled maintenance operation hard-deletes that Event; hard deletion is not a product capability.

### Error Response Shape

All API errors return:
```json
{ "statusCode": 422, "error": "ERROR_CODE", "message": "...", "details": [] }
```

All validation errors must be collected before responding — no short-circuit on first error. See `docs/prd/TeamRosterManagementBackendPRD.md` §7 for full error codes.

## Business Rules (Never Violate)

1. Active teams must have a valid `divisionId`
2. `Player.number` is unique only among active players in the same team; inactive players do not occupy jersey numbers. Enforce this in backend validation after removing `@@unique([teamId, number])` and adding an index for conflict checks.
3. `Player.number` should be `Int` (not nullable) — schema note in PRD §6.3
4. Points are never stored — always computed at request time
5. `Event.rankingOrder` is auto-assigned by backend (`max + 1`) — frontend cannot set it
6. Player management only through Team create/edit flows — no standalone player CRUD
7. Eligibility for points: `Event.status = COMPLETED`, `countsForRanking = true`, `deletedAt = null`, `archivedAt = null`
8. Different Events may use different competition formats. Represent final outcome categories with Event-specific `EventResultTag` records instead of hard-coded global placement rules.
9. `EventResultTag.sortOrder` is the display order for outcome categories. Do not introduce specific per-team rank logic unless explicitly requested.

## Design System

Dark sports-editorial theme — "Verge Dark." Near-black canvas with acid-mint as the sole hazard accent. Details in `docs/DESIGN.md`; canonical token values in `apps/frontend/src/styles/variables.css`.

- **Fonts**: Anton (display / large headlines ≥24px), Space Grotesk (body / all UI text), JetBrains Mono (stats/data/labels)
- **Colors**: near-black backgrounds (`#131313` canvas, `#1a1a1a` surface), acid-mint (`#3cffd0`) primary + success, amber (`#ffb95f`) warning, coral (`#ff6b6b`) danger
- **Shape**: role-based radius scale — 4px inputs, 16px cards/panels, 24px buttons, 20px badges; 1px borders (mostly white-alpha), no drop shadows
- **Density**: tight card padding (14–16px), information-dense layout
- Do not introduce new component libraries without explicit request
- Do not redesign existing layout patterns unless explicitly asked

### CSS Token Rules

- `apps/frontend/src/styles/variables.css` is the **canonical CSS design-token entry point**
- Use semantic tokens named by purpose (e.g., `text`, `surface`, `border`, `action`, `status` roles) — not palette names tied to the current visual design
- New or modified frontend code must not hard-code colors, font families, border radii, or repeated shadows when an appropriate token exists. Add or revise a semantic token when a genuinely shared role is missing
- Keep Mantine theme values and CSS tokens aligned — do not create an independent second theme inside page or feature code
- Reuse shared styles or components for recurring buttons, form controls, cards/panels, tables, and status indicators — do not recreate their complete visual treatment in page CSS
- Page and feature CSS should primarily own layout, spacing, responsive behavior, and genuinely local presentation. Shared visual identity belongs in tokens, the Mantine theme, or shared components
- Prefer incremental extraction based on repeated real usage — do not build speculative design-system abstractions for the MVP
- Preserve component placement, interaction logic, and responsive behavior during visual-token migrations unless the task explicitly includes those changes
- Follow `docs/UI_REDESIGN_READINESS_CHECKLIST.md` when migrating existing frontend styles or preparing a broad visual redesign

## Documentation Naming

- PRD files live in `docs/prd/`.
- PRD filenames use PascalCase with no separators.
- Split frontend and backend PRDs with suffixes:
  - `<FeatureName>BackendPRD.md`
  - `<FeatureName>FrontendPRD.md`
- `<FeatureName>` should describe the product capability, not necessarily a single database model.
- Example: `TeamRosterManagementBackendPRD.md`, `TeamRosterManagementFrontendPRD.md`.
- General project docs in `docs/` use `UPPER_SNAKE_CASE` for stable reference documents, e.g. `TEAM_POINTS_RULES.md`.
- Design and PRD feature docs should not mix kebab-case, snake_case, and PascalCase unless matching an existing file family.

## Workflow Rules

Full process is defined in `docs/WORKFLOW.md`. Summary:

**Five phases** (in order): 需求确认 → 设计文档 → 后端实施 → 前端实施 → 测试

**Three mandatory approval gates** — AI must stop and wait for user confirmation:
1. After 需求确认: user confirms §1 目标 + §2 范围 written into PRD files (not just conversation)
2. After 设计文档: user approves both completed PRDs (Backend + Frontend) including any Schema changes
3. After 测试: user accepts the test report

**Design documents** live in `docs/prd/`. Each feature produces two PRDs:
- `[FeatureName]BackendPRD.md` — scope, business rules, schema changes, API spec, validation, error codes
- `[FeatureName]FrontendPRD.md` — pages, components, API calls, UI specs

**General rules**:
- Before major schema or architecture changes: draft PRD first and wait for approval
- When uncertain about design or data model decisions: ask before implementing
- Avoid changing unrelated files — stay focused on the task
- For MVP: keep it simple, avoid over-engineering, do not implement unrequested features
- Prefer readable, maintainable code over clever abstractions
- After each task: summarize changed files and important decisions made
